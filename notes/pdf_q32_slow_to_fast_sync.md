# PDF Q32) Synchronize a signal from slow clock domain to fast clock domain

## Direct answer

The standard circuit is a **two-flip-flop synchronizer** in the **fast destination clock domain**.

```text
slow-domain signal -> FF1 -> FF2 -> fast-domain synchronized signal
                         both FF1 and FF2 use fast_clk
```

The first flip-flop may become metastable because it samples a signal from another clock domain. The second flip-flop gives the first flip-flop one fast clock cycle of settling time before the signal is used by the fast-domain logic.

Intel describes a synchronizer as a chain of destination-domain registers where the first register is driven by an unrelated or asynchronous source, and the chain gives a potentially metastable signal time to resolve before the rest of the design uses it. ([Intel FPGA metastability white paper](https://cdrdv2-public.intel.com/650346/wp-01082-quartus-ii-metastability.pdf))

## Basic circuit

```text
               fast clock domain

slow_signal ----> [ DFF 1 ] ----> [ DFF 2 ] ----> synced_signal
                    fast_clk       fast_clk
```

Names commonly used:

- `sync_1` or `meta_ff`: first stage
- `sync_2` or `sync_ff`: second stage
- `synced_signal`: safe signal used by fast-domain logic

## Verilog code

This version assumes `slow_signal` is a single-bit level/control signal from the slow domain.

```verilog
module slow_to_fast_sync (
    input  wire fast_clk,
    input  wire fast_rst,
    input  wire slow_signal,
    output wire synced_signal
);

    reg sync_1;
    reg sync_2;

    always @(posedge fast_clk or posedge fast_rst) begin
        if (fast_rst) begin
            sync_1 <= 1'b0;
            sync_2 <= 1'b0;
        end else begin
            sync_1 <= slow_signal;
            sync_2 <= sync_1;
        end
    end

    assign synced_signal = sync_2;

endmodule
```

Important point:

> Both synchronizer flip-flops are clocked by the destination clock, which is `fast_clk`.

Do not clock the first flip-flop with the slow clock and the second flip-flop with the fast clock. That would not be a proper two-flop synchronizer. The whole synchronizer chain must live in the receiving domain.

## Why two stages are needed

When `slow_signal` changes near an edge of `fast_clk`, the first flip-flop may violate setup or hold time.

So:

```text
slow_signal changes near fast_clk edge
        -> sync_1 may become metastable
        -> sync_1 gets one fast clock cycle to settle
        -> sync_2 samples the settled value
```

The second stage does not magically remove metastability with 100% certainty. It reduces the probability that metastability reaches the useful logic.

VerilogPro explains the same idea: the two-flop synchronizer uses two flip-flops in series, both clocked by the destination clock; the first stage may become metastable, while the second stage samples it after a clock period. ([VerilogPro CDC article](https://www.verilogpro.com/clock-domain-crossing-part-1/))

## Why slow-to-fast is easier than fast-to-slow

Slow-to-fast crossing is usually easier because the source signal stays stable for a long time compared to the fast clock period.

Example:

```text
slow_clk = 25 MHz   -> period = 40 ns
fast_clk = 100 MHz  -> period = 10 ns
```

If the slow-domain signal stays high for one slow clock cycle, it remains high for about four fast clock cycles. So the fast domain gets multiple chances to sample it.

That is why a normal two-flop synchronizer is usually enough for a **single-bit slow control signal** going from slow to fast.

## But the source signal should be clean

Before sending a signal across a clock domain boundary, it is good practice to register it in the source clock domain.

Good pattern:

```text
slow-domain combinational logic
        -> slow-domain register
        -> fast-domain 2-flop synchronizer
        -> fast-domain logic
```

Reason:

- combinational logic can glitch
- a glitch increases the number of transitions crossing the boundary
- more transitions means more chances for metastability

So if the source signal is generated by logic in the slow domain, first register it with `slow_clk`.

```verilog
always @(posedge slow_clk or posedge slow_rst) begin
    if (slow_rst)
        slow_signal_reg <= 1'b0;
    else
        slow_signal_reg <= slow_signal_next;
end
```

Then send `slow_signal_reg` into the fast-domain synchronizer.

## If a one-cycle pulse is needed in the fast domain

The two-flop synchronizer output is a synchronized **level**.

If the fast-domain logic needs a one-cycle pulse, create the pulse **after** synchronization:

```verilog
module slow_to_fast_pulse_sync (
    input  wire fast_clk,
    input  wire fast_rst,
    input  wire slow_signal,
    output wire synced_signal,
    output wire rising_pulse
);

    reg sync_1;
    reg sync_2;
    reg sync_2_d;

    always @(posedge fast_clk or posedge fast_rst) begin
        if (fast_rst) begin
            sync_1   <= 1'b0;
            sync_2   <= 1'b0;
            sync_2_d <= 1'b0;
        end else begin
            sync_1   <= slow_signal;
            sync_2   <= sync_1;
            sync_2_d <= sync_2;
        end
    end

    assign synced_signal = sync_2;
    assign rising_pulse  = sync_2 & ~sync_2_d;

endmodule
```

Here:

- `synced_signal` is the synchronized signal
- `rising_pulse` becomes `1` for one `fast_clk` cycle on the rising edge of the synchronized signal

## Timing condition for the signal

The input must remain stable long enough for the fast domain to sample it.

For slow-to-fast, this is usually naturally true because the slow-domain signal is wide compared to the fast clock period. But the idea is still important:

```text
source signal must be wide enough to be seen by destination clock edges
```

For a very narrow pulse, even in slow-to-fast crossing, do not rely blindly on the two-flop synchronizer. Stretch the pulse, use a toggle synchronizer, or use a handshake.

## Single-bit only warning

A two-flop synchronizer is mainly for a **single-bit control/status signal**.

Do not use this method independently on every bit of a changing multi-bit bus.

Bad method:

```text
data[0] -> two-flop sync
data[1] -> two-flop sync
data[2] -> two-flop sync
data[3] -> two-flop sync
```

Why bad?

Different bits can arrive in different fast-clock cycles. The fast domain may see a mixed bus value that never existed in the slow domain.

Intel's metastability white paper gives the same warning: asynchronous bus signals can transition on different destination clock edges, producing incorrect received bus values, so designers should use circuits such as dual-clock FIFOs or handshaking. ([Intel FPGA metastability white paper](https://cdrdv2-public.intel.com/650346/wp-01082-quartus-ii-metastability.pdf))

For multi-bit transfer, use:

- asynchronous FIFO
- valid/ready handshake
- request/acknowledge handshake
- Gray-code counter or pointer crossing

EDN also notes that conventional two-flop synchronizers are used for single-bit signals, while multi-bit synchronization can cause data incoherency unless a safe method such as Gray coding is used. ([EDN CDC synchronizer article](https://www.edn.com/synchronizer-techniques-for-multi-clock-domain-socs-fpgas/))

## Exam or interview answer

To synchronize a signal from a slow clock domain to a fast clock domain, use a two-flip-flop synchronizer in the fast clock domain. The slow-domain signal is connected to the first flip-flop. The output of the first flip-flop is connected to the second flip-flop. Both flip-flops are clocked by the fast clock. The first flip-flop may become metastable if the signal changes near the fast clock edge, but the second flip-flop samples it one fast clock cycle later, giving the first stage time to settle. The output of the second flip-flop is used as the synchronized signal in the fast domain.

## One-line memory trick

> For slow-to-fast CDC: register the source, then use two fast-clock flip-flops before using the signal.
